# AI-Supported Question Banks — Implementation Plan

> Reference: `docs/11-AI_support_question_banks.md`

## Context

Brain Gym currently relies on **manual, community-driven question creation**. Contributors write questions one-by-one through a form, which then go through a review workflow (DRAFT → PENDING → APPROVED). This limits the speed of content growth, especially for new certifications. The AI-supported question banks feature (defined in `docs/11-AI_support_question_banks.md`) introduces a full pipeline: **upload materials → chunk & index → generate (user's AI key) → critic scores → auto-publish or review → tagged question bank**.

**Phase 11.3 in the implementation plan** already envisioned AI coaching, but the specific "AI question banks" feature is not yet designed or implemented. No LLM dependencies exist in the codebase today.

## Feasibility Assessment: FEASIBLE

The feature fits naturally into the existing architecture:

- **Backend**: NestJS module system is cleanly modular — a new `ai-question-bank` module follows the exact same pattern as `flashcards`, `training`, etc.
- **Question Model**: Generated questions reuse the existing `Question` + `Choice` schema and `QuestionsService.create()`. No parallel data model needed.
- **Status Workflow**: The quality gate maps directly to existing statuses — HIGH confidence → auto-APPROVED, MEDIUM → PENDING (review queue), LOW → discarded/regenerated.
- **Review Queue**: The existing admin moderation queue (`GET /questions/queue/pending`) + community voting system already handle the review layer described in the spec.
- **Frontend**: Lazy-loaded page + shadcn/ui components + React Query hooks match all existing patterns.
- **Security**: BYOK (Bring Your Own Key) model means no platform LLM costs. Keys encrypted at rest with AES-256-GCM.
- **MCP mode**: NestJS can expose MCP-compatible endpoints for AI tools (Claude Desktop, etc.) to push structured questions into the intake API.

### What the System Already Has (Reusable)
- Question CRUD + status workflow (`QuestionsService.create()`, `updateStatus()`)
- Review queue for PENDING questions (`findPending()`)
- Community voting (up/down) — can serve as "Community Beta Pool" reviewer
- Tag auto-upsert per certification
- Gamification points on question creation
- Role-based guards (CONTRIBUTOR+)

## System Architecture (per spec)

The spec defines 6 layers. Here's how each maps to the implementation:

```
1. Material Ingestion Layer  →  SourceMaterial + SourceChunk models, upload/chunking service
2. AI Connector Layer        →  UserLlmConfig model (API Key mode) + MCP server endpoint (MCP mode)
3. Generation Pipeline       →  Cert-specific prompt templates → structured JSON output
4. Two-Pass Quality Gate     →  Critic LLM → confidence score → routing (auto-publish / review / discard)
5. Review Queue              →  Existing PENDING status + admin/community review (ALREADY BUILT)
6. Question Bank Intake      →  Existing Question model + auto-tagging (cert, domain, difficulty, source)
```

## Database Schema Changes

**New Enums:**
- `LlmProvider` — OPENAI, ANTHROPIC, GEMINI
- `GenerationJobStatus` — PENDING, PROCESSING, COMPLETED, FAILED
- `QualityTier` — HIGH, MEDIUM, LOW

**New Models:**

| Model | Purpose | Key Fields |
|-------|---------|------------|
| `UserLlmConfig` | Encrypted API keys per user/provider (Layer 2) | userId, provider, encryptedKey, modelId, isActive |
| `SourceMaterial` | Uploaded study material metadata (Layer 1) | userId, certificationId, title, originalFilename, contentType (PDF/URL/TEXT), rawContent, status |
| `SourceChunk` | Chunked passages with source metadata (Layer 1) | materialId, content, chunkIndex, pageNumber, sectionTitle, embedding (future) |
| `QuestionGenerationJob` | Audit trail + cost tracking (Layer 3-4) | userId, certificationId, domainId, provider, difficulty, questionCount, materialId (optional FK to SourceMaterial), status, promptTokens, completionTokens, qualityScores (JSON) |

**Modifications to Existing Models:**
- `Question` — add `isAiGenerated` (Boolean, default false), `generationJobId` (optional FK), `qualityTier` (optional enum), `sourceChunkId` (optional FK — traces question back to source passage)
- `User` — add relations to `UserLlmConfig[]`, `QuestionGenerationJob[]`, `SourceMaterial[]`
- `Certification`, `Domain` — add relation to `QuestionGenerationJob[]`

**File:** `backend/prisma/schema.prisma`


**Key Design: LLM Provider Interface**
```typescript
interface LlmProviderInterface {
  generateQuestions(params: GenerationParams): Promise<GeneratedQuestion[]>;
  validateApiKey(apiKey: string): Promise<boolean>;
  estimateTokens(params: GenerationParams): Promise<TokenEstimate>;
}
```
Each provider (OpenAI, Anthropic, Gemini) implements this interface. A factory instantiates the correct one based on the user's selected provider.

**Material Ingestion (Layer 1):**
- `ingestion.service.ts` handles upload → parse → chunk pipeline
- PDFs parsed via `pdf-parse` library; URLs fetched and extracted; plain text accepted directly
- Content split into chunks (~500-1000 tokens) with overlap, preserving section/page metadata
- Each chunk stored as `SourceChunk` with reference back to `SourceMaterial`
- Chunks become the grounding corpus — every generated question traces back to a specific chunk

**MCP Intake (Layer 2 — MCP mode):**
- Exposes `POST /api/v1/ai-questions/mcp/intake` endpoint
- External MCP-compatible AI tools (Claude Desktop, NotebookLM) push structured JSON questions
- Validates payload matches the same `GeneratedQuestion` schema
- Questions go through the same quality gate and intake flow as API-key-generated questions
- Authenticated via user's API token (bearer auth)

**Integration Point:** Generated questions call the existing `QuestionsService.create()` with `isAiGenerated: true` and `generationJobId` set. Quality gate determines initial status:
- Score >= 0.85 → auto-APPROVED (published to question bank)
- Score 0.60-0.84 → PENDING (enters review queue for admin/community review)
- Score < 0.60 → discarded or flagged for regeneration (not saved)

**User Flow:**
1. **Config check** — If no LLM key configured, show setup panel (API Key mode or MCP connection instructions)
2. **Material upload** — Upload PDFs, paste URLs, or enter text. System chunks and indexes the material. User can see their material library.
3. **Generation form** — Select provider, material source, certification, domain, difficulty, count (1-20). See cost estimate before generating.
4. **Preview & Edit** — Review generated questions with quality tier badges (HIGH/MEDIUM/LOW). Each question shows its source passage for traceability. Edit title, choices, explanation inline. Toggle include/exclude per question.
5. **Save** — Quality gate auto-routes: HIGH → auto-published, MEDIUM → review queue, LOW → discarded. Redirect to questions browser.

## Quality Gate — Two-Pass Pipeline (Layer 4)

Per the spec, the raw generator output goes through a **Critic LLM** (second prompt) that evaluates:
- Is the scenario realistic and exam-like?
- Are the distractors plausible (not obviously wrong)?
- Is the correct answer unambiguously supported by the source passage?
- Is the explanation accurate?

**Confidence score routing (per spec thresholds):**

| Score | Tier | Action |
|-------|------|--------|
| >= 0.85 | `HIGH` | Auto-publish to question bank (status: APPROVED) |
| 0.60 – 0.84 | `MEDIUM` | Review queue — admin or community (status: PENDING) |
| < 0.60 | `LOW` | Discarded or flagged for regeneration (not saved) |

**Generator output format** (per spec):
```json
{
  "question": "...",
  "options": ["A. ...", "B. ...", "C. ...", "D. ..."],
  "correct_answer": "B",
  "explanation": "...",
  "source_passage": "...",
  "confidence_hint": "high / medium / low"
}
```
This is mapped to our existing `CreateQuestionDto` (title, choices, explanation, etc.) before saving.

**Review Queue (Layer 5):** Both admin review and community beta pool coexist:
- Admins handle flagged edge cases via existing `/admin` moderation tab
- Trusted contributors vote up/down via existing voting system — questions hitting vote threshold auto-promote

## Security & Edge Cases

- **Key Encryption:** AES-256-GCM, encryption secret in `LLM_KEY_ENCRYPTION_SECRET` env var
- **Prompt Injection:** Source material wrapped in `<source_material>` delimiters; system prompt instructs LLM to treat it as reference data only; role markers stripped from user input
- **Response Parsing:** JSON extraction with fallback regex for markdown code fences; class-validator validation on parsed output
- **Error Handling:** Provider-specific errors (rate limit, invalid key, quota exceeded) mapped to user-friendly messages
- **Cost Control:** Token estimation before generation, 20 questions max per request, 10K char source material limit, per-user hourly rate limit
